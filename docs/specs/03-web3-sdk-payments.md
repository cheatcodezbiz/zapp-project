\"Simulate-First\" Pattern\n\nEvery write transaction in the SDK follows this mandatory pattern:\n\n```\nSIMULATE-FIRST TRANSACTION FLOW\n================================\n\n1. PREPARE\n   - Validate inputs (address format, amount > 0, etc.)\n   - Check wallet connected and on correct chain\n   - If wrong chain: prompt switch via useSwitchChain\n\n2. SIMULATE\n   - Call useSimulateContract (wagmi) with the exact parameters\n   - This calls eth_call on the node -- no gas spent\n   - If simulation reverts: parse revert reason, map to ZappError, STOP\n   - If simulation succeeds: extract gas estimate\n\n3. ESTIMATE GAS\n   - Take simulated gas and apply 20% buffer (configurable)\n   - For EIP-1559 chains: fetch baseFee + priorityFee\n   - Calculate total cost in native currency\n   - Show user: \"Estimated gas: 0.0012 ETH ($4.14)\"\n\n4. SIGN\n   - Call useWriteContract with the simulated + gas-buffered params\n   - Status: 'awaiting-signature'\n   - User approves in wallet\n\n5. WAIT\n   - Call useWaitForTransactionReceipt with confirmations: 1\n   - Status: 'pending' -> 'confirming'\n   - Show toast: \"Transaction submitted\" with explorer link\n\n6. CONFIRM\n   - Receipt received with status === 1\n   - Status: 'confirmed'\n   - Show toast: \"Transaction confirmed\"\n   - Return receipt to caller\n\n7. ERROR (at any step)\n   - Map error to ZappError with code, message, action\n   - Status: 'failed'\n   - If recoverable (gas spike, timeout): show retry button\n   - If not recoverable (revert, rejected): show error + explanation\n```\n\n### 3.5 Error Mapping\n\nThe SDK maps every known wallet/RPC/contract error to a user-friendly message:\n\n| Raw Error | ZappErrorCode | User Message | Action |\n|---|---|---|---|\n| `UserRejectedRequestError` | `TX_REJECTED_BY_USER` | \"You cancelled the transaction\" | \"Try again when ready\" |\n| `InsufficientFundsError` | higher gas\" |\n| `nonce too low` | `TX_NONCE_TOO_LOW` | \"Transaction conflict detected\" | \"Please try again\" |\n| `execution reverted: reason` | `CONTRACT_CALL_REVERTED` | Parsed reason string | `useWatchContractEvent`):\n- Subscribes to new events via WebSocket (when available) or polling (eth_getLogs every 4 seconds).\n- Transforms raw log data into typed event objects using the ABI.\n- Provides a callback-based API for UI updates.\n\n**Historical query** (via `useEventHistory`):\n- Fetches past events with pagination (block range batching to avoid RPC limits).\n- Caches results in public). Every generated dApp's `package.json` includes it as a dependency. The AI code generator is prompted to NEVER rewrite SDK functionality -- only import and use it.\n\nThe SDK is versioned with semver. Generated dApps pin to a minor version range (e.g., `^1.2.0`) so they receive patch fixes automatically but not breaking changes.\n\n---\n\n## 4. SMART CONTRACT ARCHITECTURE\n\n### 4.1 Base Contract Template\n\nAll generated contracts inherit from a base template that enforces the platform's security requirements. The inheritance chain:\n\n```\nINHERITANCE HIERARCHY\n=====================\n\n@openzeppelin/contracts-upgradeable v5.x\n\n                  Initializable\n                       |\n              +--------+--------+\n              |                 |\n    UUPSUpgradeable    OwnableUpgradeable\n              |                 |\n              +--------+--------+\n                       |\n              AccessControlUpgradeable\n                       |\n              ReentrancyGuardUpgradeable\n                       |\n              PausableUpgradeable\n                       |\n            ZappBaseUpgradeable           <-- Platform base contract\n                       |\n         +---------+---+---+---------+\n         |         |       |         |\n    ZappERC20  ZappERC721  ZappERC1155  ZappCustom\n    (token)    (NFT)       (multi-token) (arbitrary)\n```\n\n### 4.2 ZappBaseUpgradeable Contract\n\nThis is the contract that all generated implementations extend. It provides:\n\n```\nCONTRACT: ZappBaseUpgradeable\n=============================\n\nInherits:\n  - Initializable                      (@openzeppelin/contracts-upgradeable)\n  - UUPSUpgradeable                    (@openzeppelin/contracts-upgradeable)\n  - AccessControlUpgradeable           (@openzeppelin/contracts-upgradeable)\n  - ReentrancyGuardUpgradeable         (@openzeppelin/contracts-upgradeable)\n  - PausableUpgradeable                (@openzeppelin/contracts-upgradeable)\n\nRoles:\n  DEFAULT_ADMIN_ROLE   = 0x00           -- Full control, can grant/revoke roles\n  UPGRADER_ROLE        = keccak256(\"UPGRADER_ROLE\")   -- Can upgrade impl\n  PAUSER_ROLE          = keccak256(\"PAUSER_ROLE\")     -- Can pause/unpause\n\nState (ERC-7201 namespaced):\n  @custom:storage-location erc7201:zapp.base.storage\n  struct ZappBaseStorage {\n    string  dappId;                     -- Platform-assigned dApp identifier\n    string  dappVersion;                -- Semantic version of the dApp\n    uint256 deployedAt;                 -- Block timestamp of deployment\n    uint256[48] __gap;                  -- Storage gap for future base upgrades\n  }\n\nFunctions:\n  __ZappBase_init(\n    address owner,\n    string memory dappId,\n    string memory dappVersion\n  ) internal onlyInitializing\n    -- Initializes all parent contracts\n    -- Grants DEFAULT_ADMIN_ROLE + UPGRADER_ROLE to owner\n    -- Sets dappId, dappVersion, deployedAt\n\n  _authorizeUpgrade(address newImplementation)\n    internal override onlyRole(UPGRADER_ROLE)\n    -- Required by UUPSUpgradeable\n    -- Only UPGRADER_ROLE can trigger upgrades\n\n  pause() external onlyRole(PAUSER_ROLE)\n  unpause() external onlyRole(PAUSER_ROLE)\n\n  getDappInfo() external view returns (string dappId, string dappVersion, uint256 deployedAt)\n\nEvents:\n  DappUpgraded(string indexed dappId, address newImplementation, string newVersion)\n  DappPaused(string indexed dappId)\n  DappUnpaused(string indexed dappId)\n```\n\n### 4.3 UUPS Proxy Deployment Flow\n\n```\nDEPLOYMENT SEQUENCE\n===================\n\n1. COMPILE\n   Platform compiles the generated Solidity using solc (via Hardhat)\n   Outputs: implementation bytecode + ABI\n\n2. DEPLOY IMPLEMENTATION\n   Platform deployer wallet sends:\n     tx = deployContract(implementationBytecode)\n   -> implementationAddress\n\n3. ENCODE INITIALIZER\n   initData = abi.encodeWithSelector(\n     contract.initialize.selector,\n     ownerAddress,     // user's wallet address\n     dappId,           // \"dapp_abc123\"\n     dappVersion,      // \"1.0.0\"\n     ...contractSpecificParams\n   )\n\n4. DEPLOY PROXY\n   Platform deployer wallet sends:\n     tx = deployContract(\n       ERC1967Proxy.bytecode,\n       [implementationAddress, initData]\n     )\n   -> proxyAddress (this IS the dApp's address)\n\n5. VERIFY\n   Submit source code to Etherscan/Sourcify for both:\n   - Implementation contract\n   - Proxy linkage (Etherscan \"Is this a proxy?\" tool)\n\n6. TRANSFER ADMIN\n   The deployer initially holds DEFAULT_ADMIN_ROLE.\n   After verification:\n     grantRole(DEFAULT_ADMIN_ROLE, userWalletAddress)\n     grantRole(UPGRADER_ROLE, userWalletAddress)\n     renounceRole(DEFAULT_ADMIN_ROLE, deployerAddress)\n   Now the user has full control.\n   UPGRADER_ROLE also remains with platform (optional, for managed upgrades).\n```\n\n### 4.4 Upgrade Flow\n\n```\nUPGRADE SEQUENCE\n================\n\n1. AI generates new implementation code (version 1.1.0)\n2. Platform compiles new implementation\n3. OpenZeppelin Upgrades Plugin validates storage layout compatibility:\n   - No removed or reordered variables\n   - New variables only appended or in new namespaced structs\n   - Gap decremented appropriately\n4. Deploy new implementation contract\n5. Call proxy.upgradeToAndCall(newImplementationAddress, reinitData)\n   - Caller must have UPGRADER_ROLE\n   - reinitData calls reinitializer(2) if new state needs setup\n6. Emit DappUpgraded event\n7. Verify new implementation on block explorer\n```\n\n### 4.5 Storage Layout -- ERC-7201 Namespaced Storage\n\nOpenZeppelin 5.x uses ERC-7201 namespaced storage. Each contract in the hierarchy stores its variables in a struct at a deterministic storage slot:\n\n```\nSTORAGE LAYOUT EXAMPLE\n======================\n\nslot = keccak256(abi.encode(uint256(keccak256(\"zapp.base.storage\")) - 1)) & ~bytes32(uint256(0xff))\n\nZappBaseStorage at this slot:\n  [0] dappId        (string -- occupies dynamic slot)\n  [1] dappVersion   (string)\n  [2] deployedAt    (uint256)\n  [3..50] __gap     (uint256[48])\n\nThe generated contract gets its OWN namespace:\nslot = keccak256(abi.encode(uint256(keccak256(\"zapp.dapp.{dappId}.storage\")) - 1)) & ~bytes32(uint256(0xff))\n\nCustomDappStorage at this slot:\n  [0] totalMinted   (uint256)\n  [1] maxSupply     (uint256)\n  ... generated state variables ...\n  [N..N+49] __gap   (uint256[50])\n```\n\nThis eliminates storage collision risk even if the inheritance order changes between versions.\n\n### 4.6 Who Signs the Deployment Transaction?\n\n**Decision: Platform-owned deployer wallet deploys on behalf of the user.**\n\nRationale:\n- Users should not need ETH on the target chain just to deploy. They paid credits.\n- The platform controls gas pricing and can batch deployments.\n- The platform can use optimized nonce management (sequential, no conflicts).\n- After deployment, admin roles are transferred to the user's wallet address.\n\nThe deployer wallet is a hot wallet whose private key is managed via AWS KMS (see Section 5.2). It never holds significant value -- just enough gas for pending deployments.\n\n### 4.7 Contract Verification\n\n**Dual verification strategy:**\n\n1. **Sourcify** (primary): Decentralized, no API key required, supports all EVM chains. The platform submits the compiler input JSON (standard JSON input) and metadata. Sourcify performs full match verification.\n\n2. **Etherscan-family** (secondary): Covers Etherscan, Basescan, Arbiscan, Polygonscan, BscScan, etc. Requires per-explorer API keys. Uses the `hardhat-verify` plugin API programmatically (not CLI). For proxy contracts, the platform also calls the \"Is this a proxy?\" verification endpoint to link proxy and implementation.\n\nVerification is async and non-blocking. The platform queues verification jobs after deployment confirms. Status is polled and displayed on the dApp management dashboard.\n\n---\n\n## 5. DEPLOYMENT PIPELINE\n\n### 5.1 Step-by-Step Pipeline\n\n```\nDEPLOYMENT PIPELINE -- COMPLETE FLOW\n=====================================\n\n[1] CODE GENERATION (upstream -- not our domain)\n    AI generates Solidity source code\n    Output: .sol files\n\n[2] COMPILATION\n    Hardhat compilation pipeline:\n      npx hardhat compile\n    Using solc 0.8.24+ (minimum for ERC-7201 support)\n    Compiler settings:\n      optimizer: enabled, runs: 200\n      viaIR: true (for stack-too-deep prevention)\n      evmVersion: \"cancun\" (or chain-appropriate)\n    Output: artifacts/ containing:\n      - ABI (JSON)\n      - Bytecode (hex)\n      - Source map\n      - Build info (for verification)\n\n[3] STATIC ANALYSIS (parallel with compilation on second pass)\n    Slither or Aderyn scan for:\n      - Reentrancy vulnerabilities\n      - Unchecked return values\n      - Access control issues\n      - Gas optimization opportunities\n    Cost: $2.00 deducted from credits\n    Output: SecurityReport JSON\n\n[4] STORAGE LAYOUT VALIDATION (for upgrades only)\n    OpenZeppelin Upgrades Plugin:\n      validateUpgrade(proxyAddress, NewImplementation)\n    Checks:\n      - No storage collisions\n      - No removed/reordered variables\n      - Constructor usage (must be none)\n      - Delegatecall/selfdestruct absence\n    If fails: ABORT with detailed error\n\n[5] GAS ESTIMATION\n    Simulate deployment on a fork:\n      Hardhat fork of target chain at latest block\n      Deploy implementation + proxy on fork\n      Record gas used\n    Multiply by current gas price (from chain RPC):\n      estimatedCost = gasUsed * (baseFee + priorityFee) * 1.3 (buffer)\n    Convert to USD via Chainlink\n    Validate user has sufficient credits\n    Show cost confirmation to user\n\n[6] DEPLOY IMPLEMENTATION\n    Platform deployer wallet (chain-specific) sends:\n      CREATE transaction with implementation bytecode\n    Using viem's deployContract action via AWS KMS signer\n    Wait for 2 confirmations\n    Record: implementation address, deploy tx hash, gas used\n\n[7] DEPLOY PROXY\n    Platform deployer wallet sends:\n      CREATE transaction with ERC1967Proxy(impl, initData)\n    Wait for 2 confirmations\n    Record: proxy address, deploy tx hash, gas used\n    This proxy address is the canonical dApp address\n\n[8] POST-DEPLOY CONFIGURATION\n    a. Transfer ownership roles to user's wallet address\n    b. Queue contract verification (Sourcify + Etherscan)\n    c. Register contract in platform DB:\n       INSERT INTO deployed_contracts (\n         dapp_id, chain_id, proxy_address, implementation_address,\n         deploy_tx_hash, deployer_address, abi, bytecode_hash,\n         version, user_id, status, deployed_at\n       )\n    d. Set up event indexing for the new contract (see Section 6)\n    e. Deduct deployment cost from user credits\n\n[9] HEALTH CHECK\n    After deployment, verify the contract is functional:\n      - Call a read function (e.g., getDappInfo())\n      - Verify return values match expected\n      - Confirm proxy -> implementation linkage via ERC-1967 storage slot read\n    If health check fails: alert, do NOT deduct credits, investigate\n\n[10] NOTIFY USER\n     - Update dashboard: \"Deployed to {chain} at {address}\"\n     - Provide: contract address, explorer link, ABI download\n     - Enable management dashboard for this contract\n```\n\n### 5.2 Deployer Wallet Key Management\n\n**Architecture: AWS KMS with `web3-kms-signer`**\n\n```\nKEY MANAGEMENT ARCHITECTURE\n============================\n\nAWS KMS (FIPS 140-3 Level 3)\n  |\n  +-- Key: zapp-deployer-ethereum-mainnet    (secp256k1, SIGN_VERIFY)\n  +-- Key: zapp-deployer-base                (secp256k1, SIGN_VERIFY)\n  +-- Key: zapp-deployer-arbitrum            (secp256k1, SIGN_VERIFY)\n  +-- Key: zapp-deployer-optimism            (secp256k1, SIGN_VERIFY)\n  +-- Key: zapp-deployer-polygon             (secp256k1, SIGN_VERIFY)\n  +-- Key: zapp-deployer-bsc                 (secp256k1, SIGN_VERIFY)\n  ... one key per chain\n\nPrivate keys NEVER leave KMS. Signing happens inside the HSM.\n\nFlow:\n  1. Platform backend constructs unsigned transaction (viem)\n  2. Serializes transaction (RLP encoding)\n  3. Sends hash to AWS KMS for signing: kms.sign(hash, keyId)\n  4. KMS returns signature (r, s, v)\n  5. Platform assembles signed transaction\n  6. Broadcasts to chain RPC\n\nAccess control:\n  - IAM role: only the deployment Lambda/service can invoke kms:Sign\n  - CloudTrail: every signing operation is logged\n  - Key rotation: keys can be rotated without changing the Ethereum address\n    (KMS supports key material rotation; the public key/address remains stable)\n```\n\n**Why one key per chain:**\n- Nonce isolation: each chain has independent nonce sequence\n- Blast radius: if one key is compromised, only one chain is affected\n- Gas management: each deployer is funded independently on its chain\n\n### 5.3 Nonce Management\n\nNonce conflicts are the most common deployment failure. The platform uses a **centralized nonce manager** backed by Redis:\n\n```\nNONCE MANAGEMENT\n================\n\nRedis key: nonce:{chainId}:{deployerAddress}\nValue:     current nonce (integer)\n\nBefore each deployment:\n  1. INCR nonce:{chainId}:{deployerAddress}   (atomic)\n  2. Use returned value as transaction nonce\n  3. If tx fails with \"nonce too low\":\n     - Fetch on-chain nonce: eth_getTransactionCount(address, \"pending\")\n     - SET nonce:{chainId}:{deployerAddress} to on-chain value\n     - Retry with corrected nonce\n  4. If tx fails with \"nonce too high\" (gap):\n     - Fill gap with zero-value self-transfers\n     - Retry original transaction\n\nMutex: Redis SETNX lock per chain to prevent concurrent deployments\n  from racing on nonce. Lock TTL: 60 seconds.\n```\n\n### 5.4 Gas Estimation and Cost Prediction\n\n```\nGAS COST CALCULATION\n====================\n\n1. Simulate deployment on forked chain (Hardhat)\n   -> gasUsed (implementation deploy + proxy deploy)\n\n2. Fetch current gas pricing from chain RPC:\n   EIP-1559 chains: eth_feeHistory -> baseFeePerGas + maxPriorityFeePerGas\n   Legacy chains: eth_gasPrice\n\n3. Calculate:\n   gasCostWei = gasUsed * (baseFee + priorityFee) * 1.3  (30% buffer)\n   gasCostUSD = gasCostWei * nativeTokenPriceUSD          (from Chainlink)\n\n4. Add platform margin: deploymentCostUSD = gasCostUSD * 1.5  (covers gas fluctuation + platform profit)\n\n5. Display to user: \"Estimated deployment cost: $12.40\"\n   (Actual gas cost may be lower; difference is platform revenue)\n```\n\n### 5.5 Handling Deployment Failures\n\n| Failure | Detection | Recovery |\n|---|---|---|\n| Gas price spike (tx stuck) | `pending` status > 5 min | Replace with higher gas price (same nonce) |\n| Implementation deploy reverts | Receipt `status === 0` | Parse revert, fix code, retry. Refund credits. |\n| Proxy deploy reverts (init fail) | Receipt `status === 0` | Usually bad init params. Fix and retry. |\n| RPC node down | Connection timeout | Failover to backup RPC (Alchemy -> Infura -> public) |\n| Deployer out of gas funds | Insufficient balance pre-check | Auto-refill from treasury. Alert ops team. |\n| AWS KMS unavailable | KMS API timeout | Retry with exponential backoff. If persistent, failover to secondary region. |\n| Nonce conflict | \"nonce too low\" error | Resync nonce from chain (see 5.3), retry. |\n\n### 5.6 Multi-Chain Deployment Differences\n\n| Aspect | Same across chains | Differs per chain |\n|---|---|---|\n| Solidity source code | YES | -- |\n| Compiler settings | YES (mostly) | `evmVersion` may differ for newer L2s |\n| ABI | YES | -- |\n| Bytecode | YES (with same evmVersion) | -- |\n| Deployer wallet | -- | Different address per chain |\n| Gas pricing | -- | EIP-1559 vs legacy; different baseFee ranges |\n| Block confirmation time | -- | 12s (ETH) vs 2s (L2s) vs 3s (BSC) |\n| Contract verification API | -- | Different explorer API endpoints + keys |\n| RPC endpoint | -- | Different URLs per chain |\n\n---\n\n## 6. SECURITY CONSIDERATIONS\n\n### 7.1 Platform Deployer Key Security\n\n**Threat:** Compromise of deployer private key allows attacker to deploy arbitrary contracts or drain deployer gas funds.\n\n**Mitigations:**\n1. **AWS KMS (FIPS 140-3 Level 3):** Private key material never leaves the HSM. Even AWS engineers cannot extract it.\n2. **IAM least-privilege:** Only the deployment service's IAM role can call `kms:Sign`. No human has direct signing access.\n3. **CloudTrail audit:** Every KMS signing operation is logged with requester identity, timestamp, and request metadata.\n4. **Low balance:** Deployer wallets hold minimal gas (auto-refilled from treasury when below threshold). Maximum exposure is a few hundred dollars per chain.\n5. **Rate limiting:** Maximum 10 deployments per minute per chain. Anomaly detection alerts if deployment rate spikes.\n6. **Separate keys per chain:** Compromise of one chain's key does not affect others.\n7. **Key rotation:** AWS KMS supports automatic key rotation. The Ethereum address derived from the key remains stable during rotation.\n\n### 7.2 Preventing Single Point of Failure for Deployed dApps\n\n**Critical principle: Deployed dApps must survive platform shutdown.**\n\nArchitectural guarantees:\n1. **User owns admin roles:** After deployment, `DEFAULT_ADMIN_ROLE` and `UPGRADER_ROLE` are transferred to the user's wallet. The platform retains no privileged access unless the user explicitly grants it (e.g., for managed upgrades).\n2. **Contracts are on-chain:** The proxy and implementation are fully on-chain. They function regardless of the platform's status.\n3. **Open-source SDK:** The `@zapp/web3-sdk` is published on npm. Even if the platform disappears, developers can maintain their dApps.\n4. **Verified source code:** All contracts are verified on Sourcify (decentralized) and Etherscan. Anyone can read and fork the code.\n5. **No platform oracle dependency:** Generated contracts do not call back to the platform. They are fully autonomous on-chain.\n6. **User receives full source code:** The platform provides a downloadable project ZIP with all source code, compilation artifacts, and deployment scripts.\n7. **No kill switch:** The platform does not retain a `PAUSER_ROLE` or admin backdoor in deployed contracts unless the user opts in to managed upgrades.\n\n### 7.3 Rate Limiting on Deployment Endpoints\n\n```\nRATE LIMITING STRATEGY\n======================\n\nEndpoint                     | Limit            | Window  | Key\n-----------------------------|------------------|---------|----------------\nPOST /api/deploy             | 5 per user       | 1 hour  | user_id\nPOST /api/deploy             | 50 total         | 1 hour  | global\nPOST /api/credits/deposit    | 10 per user      | 1 hour  | user_id\nPOST /api/auth/siwe/nonce    | 20 per IP        | 5 min   | IP address\nPOST /api/auth/siwe/verify   | 10 per IP        | 5 min   | IP address\nPOST /api/generate           | 10 per user      | 1 hour  | user_id\n\nImplementation: Redis-based sliding window rate limiter.\nResponse on limit: HTTP 429 with Retry-After header.\n```\n\n### 7.4 Platform USDC Treasury Security\n\nThe platform accumulates USDC from user deposits. This treasury requires:\n\n1. **Multi-sig:** The treasury is a Gnosis Safe (now Safe{Wallet}) multi-sig requiring 3-of-5 signatures from founding team members for any withdrawal.\n2. **Separate from deployer wallets:** Treasury wallets and deployer wallets are completely separate key hierarchies.\n3. **Chain-local:** Each chain has its own treasury Safe.
