Database Schema\n\n### 2.1 ORM Choice: Drizzle ORM\n\n**Why Drizzle (not Prisma, not raw SQL):**\n- Drizzle schemas are TypeScript code, not a DSL. They live in our monorepo, get typechecked, and their inferred types flow directly into tRPC routers -- zero codegen step, no `prisma generate`.\n- Drizzle generates standard SQL migrations that we commit to git. No opaque migration engine.\n- Drizzle is thin -- it compiles to predictable SQL. For a platform where we need to write complex credit ledger queries and aggregations, this matters. Prisma's query engine is a black box that resists optimization.\n- Drizzle v0.45+ has mature relation support, prepared statements, and excellent PostgreSQL type coverage.\n\n### 2.2 Schema Definition\n\nBelow is the full schema expressed as Drizzle table definitions. Each table is annotated with its purpose.\n\n#### `users`\n```\nusers\n├── id              uuid        PK, default gen_random_uuid()\n├── wallet_address  text        UNIQUE NOT NULL  -- checksummed EVM address\n├── chain_id        integer     NOT NULL         -- chain used for auth (1 = mainnet, etc.)\n├── display_name    text        nullable         -- optional vanity name\n├── avatar_url      text        nullable\n├── nonce           text        NOT NULL         -- current SIWE nonce (rotates)\n├── created_at      timestamptz NOT NULL DEFAULT now()\n└── updated_at      timestamptz NOT NULL DEFAULT now()\n```\nIndex: `idx_users_wallet` on `(wallet_address)`.\n\n#### `sessions`\n```\nsessions\n├── id              uuid        PK\n├── user_id         uuid        FK → users.id ON DELETE CASCADE\n├── token_hash      text        NOT NULL        -- SHA-256 of JWT (for revocation)\n├── expires_at      timestamptz NOT NULL\n├── ip_address      inet        nullable\n├── user_agent      text        nullable\n└── created_at      timestamptz NOT NULL DEFAULT now()\n```\nIndex: `idx_sessions_user` on `(user_id)`. Index: `idx_sessions_token_hash` on `(token_hash)`.\n\n#### `credit_balances`\n```\ncredit_balances\n├── user_id         uuid        PK, FK → users.id ON DELETE CASCADE\n├── balance         numeric(20,6)  NOT NULL DEFAULT 0  -- USD-pegged credits\n├── lifetime_spent  numeric(20,6)  NOT NULL DEFAULT 0\n├── updated_at      timestamptz NOT NULL DEFAULT now()\n```\nThis is a materialized balance. The `credit_transactions` table is the source of truth; this table is a denormalized cache updated within the same transaction.\n\n#### `credit_transactions`\n```\ncredit_transactions\n├── id              uuid        PK\n├── user_id         uuid        FK → users.id ON DELETE CASCADE\n├── type            text        NOT NULL  -- 'deposit' | 'spend' | 'refund' | 'bonus'\n├── amount          numeric(20,6) NOT NULL  -- positive for deposit/refund/bonus, negative for spend\n├── running_balance numeric(20,6) NOT NULL  -- balance after this txn\n├── description     text        NOT NULL    -- human-readable (\"Deploy to Base Mainnet\")\n├── reference_type  text        nullable    -- 'project' | 'deployment' | 'generation' | null\n├── reference_id    uuid        nullable    -- FK to the related entity\n├── chain_id        integer     nullable    -- for deposits: which chain the crypto came from\n├── tx_hash         text        nullable    -- for deposits: on-chain transaction hash\n├── token_address   text        nullable    -- for deposits: which token was sent (null = native)\n├── token_amount    numeric(36,18) nullable -- for deposits: original crypto amount\n├── usd_rate        numeric(20,6)  nullable -- for deposits: conversion rate at time of deposit\n├── created_at      timestamptz NOT NULL DEFAULT now()\n```\nIndex: `idx_credit_txns_user_created` on `(user_id, created_at DESC)`. Check constraint: `running_balance >= 0`.\n\n**Design note:** Using `numeric` (arbitrary precision) rather than `float8` because we are dealing with financial data. The `running_balance` column enables fast balance lookups without summing the full history, while the full transaction log provides an audit trail. Credit deductions use `SELECT ... FOR UPDATE` on `credit_balances` to prevent double-spending under concurrency.\n\n#### `projects`\n```\nprojects\n├── id              uuid        PK\n├── user_id         uuid        FK → users.id ON DELETE CASCADE\n├── name            text        NOT NULL\n├── description     text        NOT NULL        -- user's plain-English description\n├── status          text        NOT NULL DEFAULT 'draft'\n│                               -- 'draft' | 'simulating' | 'generating' | 'compiling'\n│                               -- | 'testing' | 'deploying' | 'deployed' | 'failed'\n├── template_id     uuid        nullable FK → templates.id  -- null if custom\n├── template_params jsonb       nullable        -- filled-in template parameters\n├── chain_id        integer     nullable        -- target chain (set at deploy time)\n├── current_version integer     NOT NULL DEFAULT 1\n├── credits_spent   numeric(20,6) NOT NULL DEFAULT 0\n├── created_at      timestamptz NOT NULL DEFAULT now()\n└── updated_at      timestamptz NOT NULL DEFAULT now()\n```\nIndex: `idx_projects_user` on `(user_id)`. Index: `idx_projects_status` on `(status)`.\n\n#### `project_versions`\n```\nproject_versions\n├── id              uuid        PK\n├── project_id      uuid        FK → projects.id ON DELETE CASCADE\n├── version         integer     NOT NULL\n├── prompt          text        NOT NULL        -- the instruction that produced this version\n├── contract_code   text        NOT NULL        -- generated Solidity source\n├── contract_abi    jsonb       NOT NULL\n├── contract_bytecode text      NOT NULL\n├── frontend_bundle_key text    NOT NULL        -- S3/R2 key to the built frontend tarball\n├── test_results    jsonb       nullable        -- { passed: number, failed: number, details: [...] }\n├── ai_model_used   text        NOT NULL        -- e.g. \"claude-opus-4-6\"\n├── ai_tokens_used  integer     NOT NULL\n├── ai_cost_usd     numeric(10,6) NOT NULL\n├── created_at      timestamptz NOT NULL DEFAULT now()\n```\nUnique constraint: `(project_id, version)`.\n\n**Design note:** Storing `contract_code` directly in the database (not just a reference) is deliberate -- Solidity source files are typically 5-50KB. Storing them inline enables full-text search, diff computation between versions, and eliminates an extra S3 roundtrip for the most common read path (user viewing their code). The frontend bundle, being much larger (1-10MB), is stored in object storage with only a key reference here.\n\n#### `deployments`\n```\ndeployments\n├── id              uuid        PK\n├── project_id      uuid        FK → projects.id ON DELETE CASCADE\n├── version_id      uuid        FK → project_versions.id\n├── chain_id        integer     NOT NULL\n├── chain_name      text        NOT NULL        -- denormalized for display\n├── contract_address text       nullable        -- null until confirmed\n├── deployer_address text       NOT NULL        -- platform wallet that deployed\n├── tx_hash         text        nullable        -- deployment transaction hash\n├── block_number    bigint      nullable\n├── status          text        NOT NULL DEFAULT 'pending'\n│                               -- 'pending' | 'broadcasting' | 'confirming' | 'confirmed'\n│                               -- | 'verified' | 'failed'\n├── explorer_url    text        nullable        -- link to block explorer\n├── verification_status text    NOT NULL DEFAULT 'pending'\n│                               -- 'pending' | 'verified' | 'failed' | 'not_applicable'\n├── hosted_url      text        nullable        -- live URL for the generated frontend\n├── hosting_status  text        NOT NULL DEFAULT 'pending'\n│                               -- 'pending' | 'building' | 'deployed' | 'failed'\n├── gas_used        bigint      nullable\n├── gas_price_gwei  numeric(20,6) nullable\n├── deploy_cost_usd numeric(10,6) nullable\n├── error_message   text        nullable\n├── created_at      timestamptz NOT NULL DEFAULT now()\n└── updated_at      timestamptz NOT NULL DEFAULT now()\n```\nIndex: `idx_deployments_project` on `(project_id)`. Index: `idx_deployments_chain_contract` on `(chain_id, contract_address)`.\n\n#### `templates`\n```\ntemplates\n├── id              uuid        PK\n├── slug            text        UNIQUE NOT NULL  -- url-friendly identifier\n├── name            text        NOT NULL\n├── description     text        NOT NULL\n├── category        text        NOT NULL         -- 'defi' | 'nft' | 'dao' | 'token' | API Architecture\n\n### 3.1 Protocol Choice: tRPC\n\n**Why tRPC (not REST, not GraphQL):**\n\n1. **End-to-end type safety without codegen.** In a monorepo where the frontend and backend share `shared-types`, tRPC gives the frontend autocomplete and type errors for every API call with zero build step. REST requires OpenAPI codegen. GraphQL requires schema codegen and a resolver pattern that adds indirection.\n\n2. **This product is a single frontend consuming a single backend.** tRPC is purpose-built for this topology. GraphQL's power (multiple clients querying arbitrary field subsets) is unnecessary overhead.\n\n3. **Subscriptions via SSE.** tRPC v11 supports Server-Sent Events for subscriptions out of the box, which is exactly what we need for streaming job progress (generation, compilation, deployment) to the frontend. This avoids the complexity of WebSocket infrastructure.\n\n4. **Middleware composition.** tRPC's middleware model is cleaner than Express middleware for auth, credit checks, and rate limiting because it is typed and composable per-procedure.\n\n**Version:** tRPC v11 (current stable).\n\n### 3.2 Router Structure\n\n```\nappRouter\n├── auth\n│   ├── getNonce          query     → { nonce: string }\n│   ├── verify            mutation  → { token: string }  (SIWE signature verification)\n│   ├── session           query     → User | null\n│   └── logout            mutation  → void\n│\n├── credits\n│   ├── balance           query     → { balance, lifetimeSpent }\n│   ├── history           query     → CreditTransaction[]  (paginated)\n│   ├── depositAddress    query     → { address, chainId }  (platform deposit wallet)\n│   ├── confirmDeposit    mutation  → CreditTransaction  (after on-chain verification)\n│   └── estimateCost      query     → { credits: number }  (for a given action)\n│\n├── projects\n│   ├── list              query     → Project[]  (paginated, filterable)\n│   ├── get               query     → Project & { versions, deployments, simulations }\n│   ├── create            mutation  → Project\n│   ├── update            mutation  → Project  (name, description only)\n│   ├── delete            mutation  → void\n│   ├── generate          mutation  → { jobId }  (kick off AI generation pipeline)\n│   ├── deploy            mutation  → { jobId }  (kick off deploy pipeline)\n│   ├── upgrade           mutation  → { jobId }  (plain-English upgrade)\n│   └── jobStatus         subscription → { status, progress, output? }  (SSE stream)\n│\n├── templates\n│   ├── list              query     → Template[]  (with category filtering)\n│   ├── get               query     → Template & { paramSchema }\n│   └── preview           query     → { previewCode: string }  (show example output)\n│\n├── simulation\n│   ├── run               mutation  → { jobId }\n│   ├── getResult         query     → SimulationResult\n│   └── stream            subscription → SimSnapshot[]  (SSE stream of snapshots)\n│\n└── generation\n    ├── status            query     → GenerationJob\n    ├── getCode           query     → { contractCode, frontendPreviewUrl }\n    ├── iterate           mutation  → { jobId }  (user feedback → regenerate)\n    └── stream            subscription → { stage, progress, output }  (SSE)\n```\n\n### 3.3 Context and Middleware\n\nThe tRPC context (`context.ts`) is created per-request and contains:\n\n```typescript\ntype Context = {\n  db: DrizzleClient;\n  user: User | null;         // null for unauthenticated requests\n  redis: Redis;\n  services: {\n    auth: AuthService;\n    credit: CreditService;\n    project: ProjectService;\n    generation: GenerationService;\n    compilation: CompilationService;\n    deployment: DeploymentService;\n    hosting: HostingService;\n  };\n};\n```\n\n**Middleware stack:**\n\n1. `publicProcedure` -- base procedure, no auth required.\n2. `authedProcedure` = `publicProcedure.use(authMiddleware)` -- verifies JWT, populates `ctx.user`, returns 401 otherwise.\n3. `creditProcedure(cost)` = `authedProcedure.use(creditCheckMiddleware(cost))` -- verifies user has sufficient credits and places a hold (deducted on job completion, refunded on failure).\n\n### 3.4 Job Orchestration\n\nLong-running tasks use **BullMQ** (backed by Redis) with a pipeline pattern.\n\n**Why BullMQ:** Mature, battle-tested, excellent TypeScript support, built-in retry/backoff, rate limiting, job priorities, and parent-child job relationships. The parent-child model maps perfectly to our pipeline:\n\n```\nBUILD PIPELINE (parent job):\n  1. generate  (AI generates code)        → ~30-120s\n  2. compile   (solc compiles Solidity)    → ~5-15s\n  3. test      (Hardhat/Anvil runs tests)  → ~10-30s\n  4. deploy    (broadcast to blockchain)   → ~15-120s (depends on block time)\n  5. host      (build frontend, upload)    → ~20-60s\n```\n\nEach step is a separate BullMQ job. The pipeline coordinator creates them with dependencies so step N+1 only starts when step N succeeds. If any step fails, the pipeline is marked failed and credits are partially refunded (prorated to how far the pipeline got).\n\n**Progress updates:** Each worker updates the `jobs` table in PostgreSQL and publishes to a Redis Pub/Sub channel (`job:${jobId}:progress`). The tRPC subscription handler (`jobStatus`) listens on this channel and streams updates to the frontend via SSE.\n\n**Concurrency controls:**\n- Generation workers: 10 concurrent (limited by AI API rate limits)\n- Compilation workers: 20 concurrent (CPU-bound, limited by container pool)\n- Test workers: 10 concurrent (each needs an isolated Hardhat/Anvil instance)\n- Deploy workers: 5 concurrent (limited by nonce management complexity)\n- Host workers: 20 concurrent (mostly I/O: build + upload)\n\n---\n\n## 4. Infrastructure and DevOps\n\n### 4.1 Compilation Containers\n\n**Architecture: Kubernetes Jobs with a pre-warmed container pool.**\n\nEach compilation/test task runs in an isolated container for security (users' AI-generated code could be malicious) and reproducibility. The approach:\n\n1. **Base image:** A Docker image containing `solc` (multiple versions, 0.8.x line), `hardhat`, `anvil` (from Foundry), Node.js 22, and a minimal set of OpenZeppelin contracts pre-installed in `node_modules`.\n\n2. **Execution model:** When a compile/test job is dequeued, the worker creates a Kubernetes Job that:\n   - Mounts the generated source code via a ConfigMap or tmpfs volume (not persistent storage -- these are ephemeral).\n   - Runs compilation with a 60-second timeout.\n   - Writes artifacts (ABI, bytecode) to stdout, captured by the worker.\n   - Is automatically deleted after completion (TTL 5 minutes for debugging).\n\n3. **Security constraints:**\n   - No network access (NetworkPolicy denies all egress except DNS).\n   - Read-only root filesystem except for `/tmp`.\n   - Resource limits: 1 CPU, 2GB RAM, 10GB ephemeral storage.\n   - Non-root user.\n   - No privilege escalation.\n\n4. **Pre-warming:** A Kubernetes HorizontalPodAutoscaler keeps a pool of 5-20 warm pods (depending on load) with the base image already pulled and `node_modules` cached. This eliminates cold-start latency (~30s image pull) for the common case.\n\n**Why not Docker-in-Docker or cloud functions:**\n- Docker-in-Docker is a security hazard (requires privileged mode) and adds complexity.\n- Cloud functions (Lambda, Cloud Run) add cold start latency and make local development harder. Kubernetes Jobs give us the same isolation model in dev (via minikube/kind) and production.\n\n### 4.2 AI Routing Layer\n\nThe `ai-engine` package contains a `ModelRouter` that selects which AI model to use for each request based on task type, complexity, and cost.\n\n**Routing rules:**\n\n| Task | Primary Model | Fallback | Rationale |\n|------|---------------|----------|-----------|\n| Smart contract generation | Claude Opus | GPT-4o | Opus has superior reasoning for complex Solidity logic and security patterns |\n| Smart contract iteration/upgrade | Claude Sonnet | GPT-4o | Sonnet is faster and cheaper for incremental edits; still high quality |\n| Frontend generation | Claude Sonnet | DeepSeek Coder | React codegen is well-understood; Sonnet's speed matters for UX |\n| Test generation | Claude Haiku | DeepSeek Coder | Tests are repetitive/templated; cost optimization matters here |\n| Plain English → simulation config | Claude Sonnet | GPT-4o | Needs good reasoning but not Opus-level |\n| Contract security review | Claude Opus | GPT-4o | Security is critical; use the best model available |\n\n**Fallback logic:**\n\n```\n1. Try primary model with 30s timeout\n2. If rate limited (429) or timeout → try fallback model\n3. If fallback also fails → retry primary with exponential backoff (max 3 retries)\n4. If all retries exhausted → fail job, refund credits, notify user\n```\n\n**Cost tracking:** Every AI API call is logged with:\n- Model identifier and version\n- Input/output token counts\n- Cost in USD (computed from known per-token pricing)\n- Latency\n- Whether it was a fallback\n\nThis data feeds into the `ai_cost_usd` field on `project_versions` and into an internal analytics dashboard for monitoring spend.\n\n**Prompt management:** Prompts are stored as version-controlled files in `packages/ai-engine/src/prompts/`. Each prompt has a system template (Handlebars-style variable interpolation) and optional few-shot examples. This keeps prompts in code review, not in a database.\n\n### 4.3 Generated Code Storage and Serving\n\n**Storage: Cloudflare R2 (S3-compatible).**\n\n**Why R2:** Zero egress fees. Generated frontends will be served to end users at scale; S3 egress costs at volume are significant. R2 is API-compatible with S3, so we use the standard `@aws-sdk/client-s3` package.\n\n**Storage structure:**\n\n```\nr2://zapp-artifacts/\n├── projects/{projectId}/\n│   ├── v{version}/\n│   │   ├── contracts/\n│   │   │   ├── source.sol\n│   │   │   ├── abi.json\n│   │   │   └── bytecode.bin\n│   │   ├── frontend/\n│   │   │   └── build.tar.gz         # Built Next.js/React static export\n│   │   └── tests/\n│   │       ├── test-suite.ts\n│   │       └── results.json\n│   └── simulations/\n│       └── {simulationId}.json       # Full time-series data\n└── hosted/\n    └── {deploymentId}/               # Unpacked static files, served via CDN\n        ├── index.html\n        ├── _next/\n        └── ...\n```\n\n**Serving generated dApps:**\n\nEach deployed project gets a subdomain: `{slug}.zapp.app`. Cloudflare Workers + R2 serve the static files from the `hosted/{deploymentId}/` prefix. The Worker handles:\n- Routing all paths to `index.html` (SPA fallback)\n- Injecting the correct contract address and chain ID via environment replacement in `index.html`\n- Cache headers (immutable for hashed assets, short TTL for `index.html`)\n\n### 4.4 Deployment Service\n\nThe deployment service is the most security-sensitive component. It manages private keys that hold real funds (gas money).\n\n**Architecture:**\n\n1. **Key management:** Platform deployment wallets are HD wallets derived from a single master seed. The master seed is stored in **AWS KMS** (or GCP Cloud HSM). The deployment service never has the raw seed in memory -- it sends a signing request to KMS, which returns the signed transaction.\n\n2. **Nonce management:** Each deployer address maintains a local nonce counter in Redis (`deployer:{address}:nonce`). Before broadcasting, the service fetches the on-chain nonce and reconciles. This prevents nonce collisions when multiple deployments happen concurrently from the same deployer.\n\n3. **Gas strategy:** For each deployment:\n   - Query the RPC provider (Alchemy/QuickNode) for current base fee + priority fee.\n   - Add a 20% buffer for priority fee.\n   - Set `maxFeePerGas` to 2x current base fee (handles spikes).\n   - If the transaction is not mined within 60 seconds, resubmit with 30% higher gas (speed-up).\n\n4. **Transaction lifecycle:**\n   ```\n   Estimate gas → Sign via KMS → Broadcast → Wait for receipt → \n   Verify contract → Update deployment record → Trigger hosting pipeline\n   ```\n\n5. **Source verification:** After deployment confirms, the service submits the source code and compiler settings to Etherscan (or the chain's block explorer) for verification. This uses the Etherscan Verify API.\n\n6. **Multi-chain support:** The deployer service maintains a registry of RPC endpoints, explorer APIs, and gas oracle addresses per chain. Adding a new chain is a configuration change (add an entry to the chain registry), not a code change.\n\n### 4.5 Indexing Service\n\nAfter deployment, the platform monitors deployed contracts for:\n- Contract events (Transfer, Swap, etc.) to power the management dashboard\n- Contract balance changes\n- Failed transactions interacting with the contract\n\n**Architecture:**\n\nThe indexing service uses **viem's `watchContractEvent`** and **`watchBlockNumber`** against Alchemy/QuickNode WebSocket endpoints. For each deployed contract:\n\n1. A record is added to a `watched_contracts` table (not shown in main schema above for brevity, but part of the `indexing.service.ts`).\n2. A lightweight process subscribes to relevant events on that contract.\n3. Events are written to a `contract_events` table and surfaced on the management dashboard.\n4. If the WebSocket disconnects, it reconnects and backfills missed blocks by querying `eth_getLogs`.\n\nFor scale, if we reach hundreds of deployed contracts per chain, we switch to Alchemy's **Custom Webhooks** or **The Graph** subgraphs to offload indexing. But for launch, direct WebSocket subscriptions are simpler and sufficient.\n\n---\n\n## 5. Key Technical Decisions\n\n### 6.1 tRPC over REST or GraphQL\n\n**Decision:** tRPC v11.\n\n**Why for this product:** Zapp is a single-team monorepo with one frontend and one backend. tRPC gives us full-stack type safety with zero codegen, SSE subscriptions for streaming job progress, and composable typed middleware. The developer experience gain is enormous -- a backend engineer adds a new procedure, and the frontend engineer immediately sees it in autocomplete with full input/output types.\n\n**Risk:** If we ever need to expose a public API for third-party integrations, tRPC is TypeScript-only. Mitigation: we can expose a REST layer alongside tRPC using `trpc-openapi` for external consumers, while internal communication stays tRPC.\n\n### 6.2 Drizzle ORM over Prisma\n\n**Decision:** Drizzle ORM v0.45+.\n\n**Why for this product:** Our credit ledger queries need `SELECT ... FOR UPDATE`, aggregations, and CTEs. Prisma's query engine abstracts these away and makes optimization opaque. Drizzle compiles to predictable SQL, supports raw SQL escape hatches without leaving the type system, and does not require a codegen step. Schema changes produce standard `.sql` migration files we can review in PRs.\n\n**Risk:** Drizzle is younger than Prisma and has a smaller ecosystem. Mitigation: its SQL-first approach means we can always drop to raw queries without refactoring.\n\n### 6.3 BullMQ for Job Orchestration\n\n**Decision:** BullMQ v5.\n\n**Why for this product:** The build pipeline (generate -> compile -> test -> deploy -> host) is a sequence of long-running, failure-prone steps that need retry logic, progress tracking, and pipeline cancellation. BullMQ's parent-child job model maps directly to this. Its Redis backing is something we already need for caching and pub/sub.\n\n**Alternative considered:** Temporal.io. Temporal is more powerful for complex workflow orchestration but introduces a heavy infrastructure dependency (Temporal server cluster) and a steep learning curve. BullMQ is sufficient for our pipeline complexity and keeps the stack simpler.\n\n**Risk:** If pipeline complexity grows significantly (conditional branching, human-in-the-loop approvals, long-running sagas), we may need to migrate to Temporal. Mitigation: isolate all pipeline logic in `apps/api/src/jobs/pipelines/` so it can be refactored without touching the rest of the codebase.\n\n### 6.4 Cloudflare R2 + Workers for Hosting\n\n**Decision:** R2 for storage, Cloudflare Workers for serving generated dApps.\n\n**Why for this product:** Each generated dApp is a static site. At scale, we could serve thousands of dApps to tens of thousands of users. S3 + CloudFront would work but egress costs grow linearly. R2 has zero egress fees. Workers provide edge compute for SPA routing and contract address injection without a full server.\n\n**Risk:** Vendor lock-in to Cloudflare. Mitigation: the static files are standard HTML/JS/CSS. The Worker logic is ~50 lines. Migration to S3+CloudFront+Lambda@Edge is straightforward.\n\n### 6.5 SIWE (Sign-In With Ethereum) for Authentication\n\n**Decision:** EIP-4361 SIWE with JWT sessions.\n\n**Why for this product:** Our users are crypto-native. Wallet-based auth eliminates username/password flows, email verification, password resets, and account takeover via credential stuffing. The user's identity IS their wallet address.\n\n**Flow:**\n1. Frontend requests a nonce from the API (`auth.getNonce`).\n2. Frontend constructs a SIWE message and asks the user's wallet to sign it.\n3. Frontend sends the signature to the API (`auth.verify`).\n4. API verifies the signature, creates/finds the user, issues a JWT.\n5. JWT is stored in an httpOnly cookie, sent with subsequent requests.\n\n**Risk:** Users who lose access to their wallet lose access to their account. Mitigation: this is a fundamental property of wallet-based auth that our users understand. We can add social recovery or email-based backup later.\n\n### 6.6 HD Wallet + KMS for Deployment Keys\n\n**Decision:** Hierarchical Deterministic wallet with signing delegated to AWS KMS.\n\n**Why for this product:** The platform needs to sign deployment transactions on behalf of users. Private keys must never exist in application memory. KMS provides hardware-backed key storage with audit logging. HD derivation means we can generate fresh deployer addresses per chain without managing multiple root keys.\n\n**Alternative considered:** Each user deploys from their own wallet (the platform just broadcasts unsigned transactions for the user to sign). This is more decentralized but creates a terrible UX -- the user has to hold native tokens on the target chain for gas, sign multiple transactions, and wait. Our approach abstracts this away: user pays credits, platform handles the rest.\n\n**Risk:** Platform wallet is a single point of failure and a high-value target. Mitigation: KMS provides tamper-resistant key storage. Deployer wallets hold minimal gas balances, auto-replenished from a cold wallet. Multi-sig cold wallet for the gas treasury. Real-time monitoring on deployer wallet outflows.\n\n### 6.7 Decisions Needing Full Team Input\n\nThe following decisions should be discussed in the kickoff meeting:\n\n1. **Simulation granularity:** The Simulation Engine Lead needs to define what \"time step\" means for different protocol types. Does an AMM simulation run on block-by-block ticks, or larger time intervals? This affects the data model for `SimSnapshot` and the computational cost per simulation.\n\n2. **Frontend generation strategy:** The UX Lead and Web3 SDK Lead need to decide: does the AI generate a full Next.js app from scratch each time, or does it generate a configuration that is injected into a pre-built shell application? The latter is more reliable but less flexible. My recommendation: start with a shell approach (pre-built React app template with slots for contract interaction components) and graduate to full generation as the AI pipeline matures.\n\n3. **Multi-chain wallet abstraction:** The Web3 SDK Lead needs to decide how to handle non-EVM chains (Solana via Phantom). RainbowKit + wagmi are EVM-only. Do we use a unified wallet abstraction layer, or separate flows? My recommendation: ship EVM-only at launch, add Solana as a clearly separate code path, and evaluate universal wallet SDKs (like Dynamic or Privy) once the market signal is clear.\n\n4. **Credit pricing model:** How many credits per action? This is a product decision but has infrastructure implications (if simulation is \"free\" to encourage exploration, we need to rate-limit differently than if it costs credits). I recommend: simulations are free (rate-limited to 10/hour), generation costs 5 credits, deployment costs 10-50 credits depending on chain.\n\n5. **Upgrade semantics:** When a user says \"add a 2% fee on swaps\" to an already-deployed contract, do we: (a) generate a new contract and deploy fresh (breaking the existing address), or (b) generate an upgrade if the contract uses a proxy pattern?
